# Vistamz 内测版产品规格

> 状态：线框确认后，用于正式应用改造前的产品基线。
> 
> 核心目标：让设计师从产品参考图和可核实卖点出发，稳定产出可由运营审核的 Amazon 主图或 A+ 图片，并保留完整修改与审核记录。

## 1. 产品边界

### 要解决的问题

- 将零散的产品图片、SKU 信息和竞品启发整理成可上图的内容依据。
- 让图片方案以“画面论证卖点”为第一原则，而不是堆叠文案。
- 让设计师、运营和管理员各自只看到与其工作有关的入口和操作。
- 降低重复生图、错误提交、产品跑偏和人工沟通成本。

### 本期不解决的问题

- 不替代 ERP；只预留 SKU/卖点同步接口。
- 不承诺 AI 自动通过 Amazon 合规审核；AI 只做风险预审，运营保留最终决定权。
- 不强制固定 7 张图。主图套图可默认 7 槽，A+ 根据内容模块动态规划。
- 不让设计师在本期自行选择模型、候选数量或复杂参数，以控制成本和使用门槛。

## 2. 角色与权限

| 角色 | 可以做什么 | 不可以做什么 |
| --- | --- | --- |
| 设计师 | 创建项目、上传参考图、编辑卖点、确认方案、生图、选择候选图、局部修改、提交运营 | 审核自己提交的图片、导出最终包、管理系统设置 |
| 运营 | 查看分配给自己的待审项目、审核图片、退回并写明问题、通过图片 | 创建项目、进入设计生图台、修改原始卖点和品牌规则 |
| 管理员 | 管理账号、邀请码、品牌库、项目分配、模型策略、日志与异常任务 | 不替代设计或运营的日常审核职责 |

### 账户与分配规则

- 登录时先选择设计师或运营身份；管理员入口独立放在登录页角落。
- 一个项目必须有 `创建人`、`设计负责人` 和 `运营审核人`。
- 设计师提交整套图片后，仅被分配的运营可见。
- 邀请码仅用于内测准入：一次性邀请码绑定首次注册账号，管理员码可跨设备使用但必须记录。

## 3. 项目状态机

```text
草稿
  -> 内容待确认
  -> 图片方案待确认
  -> 设计生图中
  -> 待运营审核
  -> 退回设计修改
  -> 已全部通过
  -> 已导出
```

- 项目不会因重新生成某一张图而回退整套内容；仅对应图槽产生新版本。
- 任何退回都必须指向具体图槽、候选版本、问题类别和文字说明。
- 已通过版本不可静默覆盖。局部修改或再生成都会新建候选版本。

## 4. 创建项目与内容准备

### 必填信息

- 项目名称；有 SKU 时填写 SKU，无 SKU 时可直接创建。
- 输出类型：`Amazon 主图套图` 或 `A+ 内容`。
- 至少一张主参考图。
- 产品类别或简短产品名称。

### 参考图组

- 主参考图：锁定产品整体外观、材质、比例与主要结构。
- 其他角度图：建议上传侧面、背面、展开/收纳状态、关键细节图，用于降低模型猜测。
- 状态/功能图：仅在产品有开合、折叠、调节或转换状态时上传。
- 系统需要检测：文件可读性、透明背景边界、产品主体是否过小、图片是否过度裁切、重复图片。
- 预览容器必须完整容纳任意比例图片；不应在展示层裁掉产品。

### 卖点确认表

- 来源可为 SKU/ERP、人工输入、竞品研究或 AI 建议。
- 每条卖点包含：英文上图表达、中文依据、来源、证据等级、是否允许上图、风险说明。
- 设计师可编辑、合并和删除；不要求在此阶段逐条“审核通过”。
- 系统应提示卖点上限建议：默认最多 12 条进入方案池，由方案阶段选择和组合，而非将全部塞进图片。

## 5. 图片方案

### 方案原则

- 每张图片先定义“要证明的卖点”和“画面证据”，再定义标题或补充文案。
- 文案最少化；有标题时优先置于顶部，但 A+ 可按模块布局调整。
- 被禁止或证据不足的卖点不得用暗示方式规避。
- 同一套图必须继承同一品牌字体倾向、标题色、箭头样式、留白和视觉语气。

### 图槽要求

每个图槽包含：

- 图槽名称、输出类型、必需/可选状态。
- 主要卖点，可组合 1 至 3 个相关卖点。
- 画面证据：需要在图中看见什么，如何证明。
- 场景或背景要求、是否允许文字、标题建议。
- 尺寸、主图/A+规则、品牌规则与合规限制。

### 主图套图规则

- 第一张为白底主图：真实产品为唯一主体，无场景、无促销文案、无 Logo。
- 主产品建议占有效画面 80% 至 90%，默认目标 85%，同时保留完整边界和自然阴影空间。
- 非第一张可用真实场景或品牌背景，必须优先通过画面证明卖点。

### A+ 规则

- 不要求第一张白底图。
- 可组合卖点、补充主图套图未覆盖的允许卖点，并使用更丰富的叙事和模块化画面。
- 可使用已配置品牌 Logo；Logo 不进入普通主图套图。
- 尺寸、模块数量和标题位置由方案决定，不复用主图规则。

## 6. 生图与候选图

### 固定生成模式

| 模式 | 行为 | 目的 |
| --- | --- | --- |
| 当前图多张 | 仅当前图槽固定生成 3 张候选图 | 对重点图做构图、真实感和文案表现比较 |
| 整套各一张 | 每个待生成图槽固定生成 1 张 | 快速得到完整初稿，降低 API 压力 |
| 局部修改 | 基于当前候选图和选区生成 1 张新候选图 | 保留已满意的部分，仅处理局部问题 |

### 局部修改

- 默认使用智能选择；提供画笔、套索、矩形与加选/减选。
- 用户必须填写“只修改什么”和“必须保持不变什么”。
- 每次局部修改生成新候选，不替换原候选；可随时回退。
- 系统将选区、修改指令、来源候选和生成结果写入版本记录。

### 生成任务体验

- 每个任务显示排队、生成中、AI 预审中、成功、失败、可重试。
- 失败说明必须可理解，例如“模型未返回图片”“参考图读取失败”“请求超时”。
- 同一图槽的重复点击要去重；每次生成使用任务 ID，避免浏览器刷新后丢失状态。
- 系统限制项目级并发与每日额度，并记录图片、模型和成本估算。

## 7. AI 预审与人工审核

### AI 预审检查

- 产品一致性：结构、颜色、材质、关键部件、参考图角度。
- 图像基础质量：清晰度、裁切、比例、背景、物理逻辑。
- 卖点可视化：图中是否存在支撑该卖点的视觉证据。
- 文字与品牌：可读性、乱码、卖点准确性、禁止的色号/提示词泄漏、Logo 使用范围。
- 输出类型规则：主图白底、主体占比、A+ 品牌模块规则等。

AI 给出“可进入人工审核 / 需关注 / 建议重生”以及原因；不得自动批准最终图片。

### 人工审核

- 运营看到图片预览、对应卖点、画面证据、原始参考图和 AI 预审结果。
- 运营可：通过、退回修改、附加说明。
- 退回原因至少包含：产品不一致、卖点或文字错误、物理/比例/场景问题、排版或品牌规范问题、合规风险。
- 所有必需图槽均有运营通过版本时，项目才进入“可导出”。

## 8. 品牌库

### 8.1 设计原则

- 品牌库不是“把一串色号塞进提示词”，而是每次生图都可执行、可审计的视觉规则。
- 所有项目在创建时可选择品牌或“不套用品牌”；选择后锁定品牌版本快照。
- 规则分为硬约束与软偏好：硬约束未满足即不能进入人工审核；软偏好用于引导模型和人工判断，不因一次偏离而自动阻断。
- 主图套图与 A+ 继承同一品牌基调，但使用不同的规则集；Logo 只属于 A+。

### 8.2 品牌配置字段

| 规则组 | 字段 | 约束等级 | 用途 |
| --- | --- | --- | --- |
| 基本信息 | 品牌名、适用市场、版本号 | 硬 | 项目归属与历史追溯 |
| 色彩角色 | 主色、辅助色、中性色、强调色，各用 HEX 和建议比例 | 硬/软 | 用“用途”指导背景、标题、按钮、线条和点缀，而不是把色号展示给模型用户 |
| 标题系统 | 标题色、大小倾向、大小写规则、简洁度、标题默认位置 | 硬/软 | 保持整套图标题一致；A+ 可按模块改变位置 |
| 图形元素 | 箭头样式、标注线样式、图标风格、圆角倾向 | 硬/软 | 统一功能说明图的视觉语言 |
| 画面语气 | 真实/克制/温暖/技术感等、背景类型、光线、留白程度 | 软 | 指导场景图和 A+ 的审美方向 |
| 品牌资产 | Logo、可选品牌示例图、禁用图案或元素 | 硬/软 | Logo 仅 A+；示例图作为视觉参考，不应直接抄图 |
| 输出限制 | 主图是否禁 Logo、A+ 是否允许 Logo、文字语言、禁止出现的内容 | 硬 | 防止跨输出类型的规则泄漏 |

### 8.3 配色的正确表达

- 管理员输入 HEX 色号和使用比例，但系统内部应把它们映射为语义角色，例如“深墨绿作为标题和线条主色，浅米白作为大面积背景，橙色仅作为小面积强调”。
- 使用比例是构图预算，不是图片中的视觉对象；禁止生成色卡、`#HEX`、百分比或“品牌指南”面板。
- 每种颜色需标注可用范围：`背景`、`标题`、`箭头/标注`、`按钮/强调`、`不可使用`。单纯总比例无法指导模型，也无法让预审判断偏差。
- 色彩应允许容差。AI 图片中的光线、材质和压缩会造成像素偏移；质量检查以“色相/明度范围和使用位置”判断，不要求逐像素相同。

### 8.4 如何让生图尽可能严格遵循品牌

1. **项目锁定品牌快照**：创建项目时选择品牌；生成任务保存当时完整品牌版本，之后修改品牌库不回写历史项目。
2. **结构化提示词编译**：系统将品牌规则拆进固定字段：输出类型、颜色角色、标题规则、箭头规则、Logo 许可、禁止项、画面语气。不能依赖人工复制一段品牌描述。
3. **提供品牌视觉参考**：除 Logo 外，允许管理员上传 2 至 5 张已确认的品牌画面示例，作为风格参考。没有示例图时，模型只能做到近似风格，无法保证审美一致。
4. **按输出类型编译规则**：主图第一张强制白底、禁 Logo、禁花哨品牌背景；A+ 才允许标题色、Logo、品牌背景和更多图形元素。
5. **生成后自动验收**：预审检查标题位置、文字可读性、Logo 范围、箭头样式、主色出现位置、提示词泄漏、主图白底和主体占比。不满足硬约束则标记“需重生”，而不是让运营从头发现。
6. **人工审核保留品牌判断**：AI 可检查可量化项，但“是否像这个品牌”仍需要运营/品牌负责人最终确认。审核界面应显示本项目的品牌规则摘要和示例图。

### 8.5 严格性的真实边界

- 图像模型可以被强约束，但无法仅凭文字保证精确字体、精确 HEX 像素值或百分之百相同的箭头几何。
- 因此系统的“严格”应来自：结构化输入、品牌示例图、硬约束预审、失败重生、人工最终判断这五层，而非承诺一次生成必然完美。
- 当前坚持让模型直接完成画面中的文字排版是合理的，因为文字和构图会更协调；代价是文字质量波动。第一阶段应严格检测乱码、错误卖点和位置偏差并自动拦截。若内测数据证明不稳定，再增加可选的精确文字修订能力，而不是现在强行二次叠字。

### 8.6 本期品牌库优化优先级

**P0：正式改造前加入**

- 品牌版本与项目快照。
- 色彩角色和可用范围，不只存色号与比例。
- 标题色、箭头样式、Logo 的主图/A+范围限制。
- 硬约束预审：Logo 越界、主图非白底、色号/比例泄漏、禁用元素。

**P1：内测中加入**

- 品牌示例图组和生成时引用。
- 统一图标/标注线/圆角等图形元素的可选样式。
- 运营审核中的品牌规则摘要与示例对照。
- 品牌规则变更历史、克隆品牌和安全删除。

**P2：数据验证后加入**

- 为不同市场、品类或季节维护品牌子风格。
- 通过历史审核通过图自动沉淀品牌优秀案例。
- 对颜色占比、标题位置、箭头形状进行更细的自动视觉评分。

### 8.7 品牌库结论

现有字段是好的起点：HEX + 比例、Logo 仅 A+、标题颜色、箭头样式都应保留。最需要优化的是增加“颜色的使用角色、品牌版本快照、视觉示例图、硬/软规则区分和自动验收”。做到这一步，品牌库才会真正约束生成结果，而不只是影响一段提示词。

## 9. 数据、日志与导出

### 必须持久保存的数据

- 用户、角色、邀请码使用记录。
- 项目、分配关系、参考图、卖点、图片方案、图槽状态。
- 生成任务、提示词版本、模型、候选图、局部修改选区和生成版本链。
- AI 预审、运营审核、退回意见、导出记录、品牌规则快照。

### 日志要求

- 记录登录、项目创建、上传、编辑、生成、失败、重试、审核、导出和管理员操作。
- 每条日志包含时间、用户、项目、对象、动作、结果和关联任务 ID。
- 日志用于排错和审计，不在普通设计/运营界面堆砌展示。

### 导出

- 仅导出运营已通过的最终候选图。
- ZIP 按项目和输出类型命名，图片文件名包含图槽序号与简短名称。
- 导出前再次检查图槽是否完整、文件可访问、尺寸匹配目标类型。

## 10. 内测验收标准

- 不同产品类别均可完成从创建到导出的完整闭环。
- 设计师不能进入运营审核台；运营不能进入设计生图台。
- 一个项目的图片、卖点和审核记录不会被另一个项目继承或覆盖。
- 生成失败可见、可重试且不丢失项目上下文。
- 运营可在一个页面完成图片预览、问题判断和通过/退回。
- 仅全套必需图片均通过时可导出。
- API Key 不会暴露给浏览器或普通用户。

## 11. 生图逻辑评估与升级路线

### 当前已合理的设计

- 用参考图组锁定产品，而不是纯提示词生图。
- 用卖点确认表作为内容依据，而不是让模型自由编卖点。
- 主图与 A+ 在方案阶段分流，避免后续规则冲突。
- 当前图 3 张、整套各 1 张的固定策略能控制 API 成本。
- 先选候选、再整套提交运营，减少设计师和运营的往返。
- 局部修改保留原版本，适合解决“整体满意但局部不对”的真实场景。

### 上线前必须升级（P0）

1. **方案证据映射**：每个图槽必须有“卖点 -> 画面证据 -> 允许文案”的结构化字段；否则跨品类时容易生成漂亮但没有论证力的图。
2. **参考图质量闸门**：上传后识别主体边界、透明 PNG 异常、裁切风险和角度覆盖度；不合格时在生成前提示补图。
3. **异步任务与重试**：生图、预审、压缩、导出都进入任务队列，支持进度、失败分类、重试和幂等，避免多人内测时状态混乱。
4. **项目级产品锁**：为每个项目生成并持久化产品身份摘要（材质、颜色、结构、关键部件、禁改项），每次生图和预审都带入。
5. **图槽动态化**：保留“主图套图默认 7 槽”，但允许按产品和 A+ 模块创建、删除、排序可选图槽。

### 应尽快升级（P1）

1. **真遮罩局部修改**：优先接支持 mask/inpainting 的模型接口。若当前模型不支持，明确显示“区域修改建议模式”，不可伪装为精确编辑。
2. **双层质量门**：第一层视觉一致性与物理逻辑；第二层 OCR/文字、品牌规范、主图/A+尺寸和主体占比。AI 预审结果必须按输出类型采用不同规则。
3. **提示词合同**：将产品锁、卖点、画面证据、禁区、品牌规则和输出规格拆为结构化片段再组装，禁止依赖一段不断变长的自由提示词。
4. **候选对比工具**：在同一图槽内支持原图/候选/局部修正版的并排对比，帮助设计师快速决定。
5. **成本与并发保护**：每日项目预算、队列优先级、取消任务、缓存已完成结果，避免误操作重复计费。

### 可在内测数据足够后再做（P2）

1. 不同模型的 A/B 跑图与质量评分，按品类选择推荐模型。
2. 图中文字的专门质量评分。当前优先让模型直接完成排版；若稳定性不足，再提供“可选的精确文字修订”能力，不强制二次叠字。
3. ERP 自动同步、批量项目创建与报表。
4. 更细的协作角色、通知和 SLA。

### 生图结论

当前逻辑适合作为小范围内测的生成闭环，但要升级为可持续多人使用的系统，P0 是前置条件。其中最关键的不是增加更多模型，而是让每次生图都拥有可靠的产品身份、画面证据和可恢复的任务记录；这样模型质量才可以被稳定地比较和迭代。
