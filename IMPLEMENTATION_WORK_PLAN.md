- 主图方案现支持 1–9 个图槽、A+ 支持 4–7 个模块；设计师可添加可选图槽、删除、前后排序，结构变化会清理旧候选与审核状态，避免错图槽提交。
- 多角度参考选择已修正：主图第 1 槽只锁定主参考图，其余图槽会按场景、状态、结构、细节或尺寸类型组合相应参考图，不再只把同一张正面图传给模型。
- 跨设备恢复 A+ 项目时的输出类型映射已修正，不会被错误恢复成主图规则。
- 关键权限、品牌版本、任务队列、资产存储、多语言与图槽合同自动测试共 34 项，正式构建通过。
- 阶段 3 已完成：卖点支持编辑、合并、删除并显示来源与风险；参考图加入重复画面检测；产品锁加入关键部件和不可改变项；品牌色加入用途、适用范围与保存硬校验。
- 多语言输出契约已接通：项目独立保存目标 Amazon 站点与图片文案语言，图片方案、生图、局部修改、失败重试和 AI 预审统一遵循同一语言；旧项目默认 Amazon US / 英文。
- 阶段 4 已完成：主图与 A+ 使用独立动态规划；每个图槽保存卖点到画面证据映射、允许文案、场景必须项、品牌规则、输出规格和合规边界，并贯通生图与 AI 预审。

### 已接线，发布前仍需配置或补齐

- 资产存储已支持 Cloudinary 与 S3 兼容服务；Cloudinary 本地真实图片上传、读回、签名地址和删除已通过，健康页返回 `durable: true`。

## 0. 改造准备与技术决策

### 目标

冻结当前可运行版本，确定内测期的数据和文件保存方式，避免之后继续把多人数据写在浏览器本地。

### 工作项

- 建立可回退的发布标签和数据库迁移策略。
- 确认正式内测架构：前端、Node API、数据库、图片对象存储、任务队列。
- 保持一把公司管理的图像 API Key，只留在服务端。
- 明确环境：开发、预览、内测；每个环境使用独立的密钥和数据。
- 制定现有 `localStorage` 草稿的迁移与保留策略。

### 推荐默认决策

- 前端：继续 Vercel。
- 后端：继续 Node API 服务。
- 数据库：Postgres。
- 图片和导出包：对象存储，不依赖 Render 临时磁盘。
- 任务：后端任务表 + worker；先不引入过重的分布式系统。

### 完成标准

- 不再依赖浏览器本地数据来保存正式项目。
- 项目、图片和日志有明确持久化位置。
- 本地、预览、内测环境互不污染。

## 1. 账号、权限、项目数据与日志底座

### 目标

让每一个动作都知道“谁做的、属于哪个项目、交给谁处理”，这是多人内测的第一前提。

### 工作项

- 接入真实登录与账号体系。
- 实现三种身份：设计师、运营、管理员。
- 一次性邀请码绑定首个账号；管理员码受限使用并留痕。
- 建立用户、项目、项目分配、邀请、品牌、日志等数据表。
- 项目必须保存创建人、设计负责人和运营审核人。
- 接入统一事件日志：登录、创建、上传、生成、失败、重试、审核、导出、管理操作。
- 在服务端实现权限拦截，而不是只靠前端隐藏入口。

### 完成标准

- 设计师无法访问运营审核 API；运营无法访问设计生图 API。
- 运营只看到分配给自己的项目。
- 管理员可定位任意项目的操作历史和失败任务。

## 2. 正式应用信息架构与页面骨架

### 目标

将已确认的线框转化为正式页面，但先不接复杂生图能力，确保新手一眼知道自己该做什么。

### 工作项

- 替换旧式后台导航为登录分身份、项目中心、设计工作台、运营审核台和管理员入口。
- 侧边栏可收起；创建项目只保留项目中心一个入口。
- 每个身份仅显示自己的下一步任务，不展示无关状态和技术信息。
- 落地固定底部行动条、步骤进度、空状态、加载状态、失败状态和操作反馈。
- 将已确认的线框规范整理成可复用组件，避免每页样式漂移。

### 完成标准

- 新用户从登录进入后，30 秒内能说出当前要做什么。
- 设计师和运营的导航、按钮和页面入口完全隔离。
- 页面在常见桌面尺寸下无换行错位、溢出或按钮遮挡。

## 3. 项目资料、卖点和品牌快照

### 目标

在生图前把“产品是什么、能说什么、属于哪个品牌”变成可靠输入。

### 工作项

- 实现有 SKU / 无 SKU 两种创建模式。
- 参考图组支持主参考图、其他角度、状态/功能、细节图。
- 上传时检查图片可读性、主体边界、过度裁切、透明 PNG 异常和重复图。
- 建立产品锁：材质、颜色、结构、关键部件、禁改项与参考图绑定。
- 简化卖点确认表：可编辑、合并、删除、来源与风险可见；不要求逐条审核状态。
- 选择品牌后保存品牌版本快照到项目。
- 品牌库先实现 P0：颜色角色、标题色、箭头样式、Logo 使用范围、品牌版本。

### 完成标准

- 新项目不会继承其他项目的图片、卖点、方案或品牌规则。
- 产品参考不足时，系统在生图前明确提示需要补什么图。
- 品牌库更新不会影响已创建项目的历史任务。

## 4. 动态图片方案与画面证据

### 目标

让方案从“固定 7 张卡片”升级为“根据产品与输出类型规划的图槽”，每张图都有可判断的目的。

### 工作项

- 在方案前选择输出类型：主图套图或 A+。
- 主图套图默认 7 槽，但支持增删、排序和标记可选图槽。
- A+ 按模块规划，不继承第一张白底规则，可组合相关卖点。
- 每个图槽保存：主要卖点、画面证据、允许文案、场景、品牌规则、尺寸和合规限制。
- 方案生成时引用产品锁、卖点池与品牌快照。
- 设计师在此阶段只确认方向或要求重做当前图槽方案，不进入运营审核。

### 完成标准

- 任一图槽都能回答：“要证明什么、画面里必须出现什么、哪些不能出现”。
- 同一项目不同输出类型不会混用尺寸、Logo 或第一张图规则。
- 用户可以理解为什么是这些图槽，而不是只看到 AI 起的标题。

## 5. 生图任务、候选图和局部修改

### 目标

以成本可控、结果可恢复的方式，完成从方案到可审候选图的生成。

### 工作项

- 将品牌、产品锁、卖点、画面证据和输出规格编译为结构化提示词合同。
- 实现两种固定模式：当前图槽生成 3 张；整套各图槽生成 1 张。
- 生图任务进入后端队列，显示排队、生成中、AI 预审中、成功、失败和重试。
- 添加任务去重、取消、并发限制、项目日额度与成本记录。
- 候选图支持并排比较、选择、保留版本和重新生成。
- 局部修改先对接支持 mask/inpainting 的模型；不支持时明确作为“区域修改建议”，不可伪装为精确编辑。
- 选定候选自动标记图槽为“已选”；必需图槽全部选定后才允许整套提交运营。

### 完成标准

- 浏览器刷新或任务失败后，项目能恢复到准确的任务状态。
- 每次候选、局部修改和再生成均可追溯到来源和指令。
- 设计师无法误选候选数量造成额外 API 消耗。

## 6. AI 预审、运营审核与交付

### 目标

让 AI 拦截明显错误，让运营快速判断业务、合规和品牌风险，并形成明确回改闭环。

### 工作项

- AI 预审分类型规则：产品一致性、物理逻辑、主体占比、背景、OCR/乱码、卖点可视化、品牌和输出类型规则。
- 主图第一张检查白底、完整产品和 80% 至 90% 主体占比；A+ 使用独立规则。
- 审核页展示候选图、原始参考图、卖点、画面证据、品牌摘要和 AI 预审。
- 运营可通过或退回；退回绑定图槽、候选版本、问题分类和说明。
- 设计端收到退回后，只重做相应图槽并保留历史版本。
- 所有必需图槽存在运营通过版本后，才可导出。

### 完成标准

- AI 预审不再出现“显然变形却放行”的情况；不确定时宁可提示人工关注。
- 运营无需切换页面或反复滚动即可完成单图审核。
- 导出 ZIP 只包含每个图槽的运营通过最终版本。

## 7. 内测质量验证、部署与复盘

### 目标

用真实产品和真实用户验证闭环、成本和质量，不在没有数据前过度扩展功能。

### 工作项

- 选择至少三种产品类型，包含不同结构、材质和场景需求。
- 每种产品分别测试主图套图和 A+ 内容。
- 记录任务成功率、平均生成时长、每套成本、AI 预审命中率、人工退回原因和最终可用率。
- 运行小规模设计师/运营测试，收集操作路径和困惑点。
- 整理失败任务日志，按“产品一致性、文案、物理逻辑、品牌、平台稳定性”归因。
- 每周只优先处理影响闭环或成功率最高的问题。

### 完成标准

- 三类产品均可完成创建、生成、审核与导出。
- 内测者不依赖口头教学也能完成主要流程。
- 有明确的下一版本优先级，而不是根据单个反馈随机加功能。

## 每阶段的协作规则

- 每完成一个阶段，先在预览环境演示真实流程，再决定是否进入下一阶段。
- 不在 UI 未稳定前插入无关新功能；不在数据底座未完成前扩大内测人数。
- 每项功能都要同时给出：用户操作、权限、失败状态、日志事件和验收方式。
- 所有生成策略变化需记录模型、提示词版本、品牌版本和项目版本，便于复现问题。

## 第一轮建议执行顺序

1. 阶段 0：确认数据库和对象存储方案，并备份当前应用。
2. 阶段 1：账号、权限、项目分配、日志。
3. 阶段 2：将确认的线框落进正式应用。
4. 阶段 3：项目资料、卖点、品牌 P0 与项目快照。
5. 阶段 4：动态方案和画面证据映射。
6. 阶段 5：任务队列、生图、候选与局部修改。
7. 阶段 6：预审、运营审核、导出门槛。
8. 阶段 7：三类产品的内测验证与复盘。

这份顺序的原则是：先让多人协作的数据和责任有归属，再做生成体验；先保证每次结果可恢复、可审计，再追求更多模型和更复杂的美学能力。
